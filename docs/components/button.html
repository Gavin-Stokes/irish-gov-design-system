Button

Buttons trigger actions or navigate between screens. Each screen should have one clearly dominant primary action.

Component WCAG 2.1 AA

Variants

Primary

The most prominent action on a screen. Fill: #00875F. Limit to one per screen.

Primary (dark)

Alternate primary for home screen context. Fill: #004D44. Used when the standard green primary would clash with a green background.

Secondary

A secondary action alongside a primary. Transparent background, #004D44 border and text.

Tertiary

Low-priority, text-only action. Underlined, no background. Use sparingly.

Back link

Navigates to the previous screen. Always names the destination — never "Back".

When to use

  • Use a primary button for the main action on a screen (submitting a form, moving to the next step).
  • Use a secondary button for optional or alternative actions alongside a primary (cancel, save draft).
  • Use a tertiary button for low-priority actions that don't need visual weight (return to dashboard, learn more).
  • Use a back link for navigation to a previous screen — not for form submission or actions.

When not to use

  • Do not use multiple primary buttons on the same screen — choose one dominant action.
  • Do not use buttons for navigation to external resources — use a text link with the open_in_new icon instead.
  • Do not disable buttons. If an action can't proceed, show inline validation explaining why rather than disabling the submit button.

Do and don't

check_circle
Do use sentence case and action-oriented copy: "Apply now", "Submit application".
cancel
Don't use ALL CAPS, vague labels like "Submit", or Title Case.
check_circle
Do pair a primary with a secondary for optional alternate actions.
cancel
Don't stack two primary buttons — one must be dominant.
info
Buttons are among the most frequently interacted-with elements in the application. Getting them right for keyboard users and screen readers is critical.

Accessibility checklist

  • check_circle Use a native <button> element for actions and <a href> for navigation. Never use <div> or <span> as a button. 4.1.2 A
  • check_circle Every button must have a discernible text label — either visible text or an aria-label. Icon-only buttons require aria-label="Description". 1.1.1 A
  • check_circle Minimum touch target: 44×44px. Standard buttons are 48px tall; large CTAs are 58px. Full-width on mobile. 2.5.5
  • check_circle Focus ring must always be visible: outline: 3px solid #FFBF47; outline-offset: 2px. Never suppress with outline: none. 2.4.7 AA
  • check_circle White text on #00875F (Action Green) achieves a 4.5:1 contrast ratio — the minimum for AA. Verify at the smallest text size used on the button. 1.4.3 AA
  • check_circle For async actions (form submission), use aria-busy="true" on the button during processing and update an aria-live region with the result. 4.1.3 AA
  • check_circle Back links must be <a> elements (navigation), not <button> (action). They link to a specific screen, not browser history. 4.1.2 A
  • check_circle Decorative trailing icons (arrow_forward) must be aria-hidden="true" to prevent screen readers announcing "arrow forward" after the button label. 1.1.1 A

HTML

HTML
<!-- Primary button -->
<button class="btn btn--primary" type="button">
  Apply now
  <span class="material-icons" aria-hidden="true">arrow_forward</span>
</button>

<!-- Primary — full width, large -->
<button class="btn btn--primary btn--full btn--lg" type="submit">
  Submit application
  <span class="material-icons" aria-hidden="true">arrow_forward</span>
</button>

<!-- Secondary -->
<button class="btn btn--secondary" type="button">
  Save and return later
</button>

<!-- Tertiary -->
<button class="btn btn--tertiary" type="button">
  Return to dashboard
</button>

<!-- Back link -->
<a href="/home" class="back-link">
  <span class="material-icons" aria-hidden="true">chevron_left</span>
  Home
</a>

<!-- Icon-only button -->
<button class="btn" type="button" aria-label="Notifications">
  <span class="material-icons" aria-hidden="true">notifications</span>
</button>
