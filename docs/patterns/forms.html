Multi-step forms

Guidance for designing and building long, multi-step forms that citizens can complete with confidence — including error handling, progress, and save states.

Pattern WCAG 2.1 AA

Structuring a multi-step form

Break long forms into logical sections — typically 5–8 fields per screen. One topic per screen: personal details, employment, income, housing requirements. This approach:

  • Reduces cognitive load — citizens focus on one area at a time.
  • Improves completion rates — shorter screens feel more achievable.
  • Makes errors easier to understand and fix — errors are isolated to their section.
  • Enables partial saves — citizens can leave and return without losing progress.

Before you begin screen

Always include a "Before you begin" screen at the start of a multi-step form. It must include:

  • A list of all sections the citizen will complete
  • The estimated time to complete ("approximately 10 minutes")
  • What information the citizen needs to have ready (PPSN, income details, address)
  • A data notice explaining how the submitted information is used and stored
  • A clear primary CTA: "Start application"

Step counter

Display a text step counter above the page <h1> on every screen in the flow: "Step 2 of 5". Do not use a visual progress bar alone — it is not perceivable by screen readers and conveys no information to citizens who cannot see it.

One question per page (where possible)

Where a single question is complex (e.g. employment status with a conditional follow-up), consider giving it its own screen rather than grouping it with other questions. The GOV.UK "one thing per page" principle reduces errors and improves completion rates in government form contexts.

Conditional questions

Reveal conditional fields only when triggered by a previous answer. The employer name field should only appear when "Employed" is selected. Use JavaScript to show/hide the field — and move focus to the newly revealed field.

Validation

  • Validate on blur — when the user leaves a field, validate it and show any error immediately.
  • Validate on submit — show all errors when the "Continue" button is pressed.
  • Never validate on keypress — showing errors while the citizen is still typing is disruptive and creates false error states.
  • Error summary — on failed submission, show a summary at the top listing all errors. Each error item links to its field.

Save and return

Include a "Save and return later" secondary button on each section screen. On return:

  • The Before You Begin screen shows completed sections with a "Complete" label.
  • The citizen can resume from any incomplete section.
  • Pre-filled data is retained.
  • Confirmation of what was saved should appear as a notification on return.

Review screen

Before submission, show a full review screen. Each section is summarised with an "Edit" link. Edit links include the section name in their accessible label: "Edit personal details", not just "Edit".

Success screen

The confirmation screen must include:

  • A unique reference number (e.g. SH-2026-00142)
  • Clear next steps ("You will hear from your local authority within 10 working days")
  • An option to print or save the confirmation
  • A return to dashboard link
warning
Multi-step forms introduce specific accessibility risks: focus management between steps, error announcement, and session timeout. Each of these must be explicitly handled.

Accessibility requirements

  • check_circle When the citizen moves to a new step, move focus to the new <h1> or a skip target at the top of the new content. Do not leave focus on the "Continue" button from the previous screen. 2.4.3 A
  • check_circle Update the <title> element when the step changes: "Employment — Social Housing Application". This ensures screen reader users know they are on a new screen. 2.4.2 A
  • check_circle Error summaries must receive focus immediately after a failed submission attempt. Use tabindex="-1" on the summary and call .focus() in JavaScript. 3.3.1 A
  • check_circle Each error in the summary must be a link that jumps to the relevant field: <a href="#field-id">Error message</a>. Moving focus to the field is more helpful than scrolling alone. 3.3.3 AA
  • check_circle Conditional fields revealed by JavaScript must be focusable when shown. Move focus to the newly revealed field using .focus() after the DOM update. 2.1.1 A
  • check_circle If the session will expire, warn the citizen at least 20 seconds before timeout with an aria-live="assertive" announcement. Provide options to extend the session or save progress. 2.2.1 A
  • check_circle The review screen's edit links must have unique accessible labels. "Edit personal details" is unambiguous; "Edit" alone is not (WCAG 2.4.6 requires descriptive headings and labels). 2.4.6 AA
  • check_circle The success screen reference number should be announced via aria-live="polite" or be clearly within the reading order of the page so screen reader users do not miss it. 4.1.3 AA
  • check_circle Do not use CAPTCHA or other time-limited challenges in citizen-facing government forms. They create significant barriers for users with cognitive and visual disabilities. 1.1.1 A / 2.2.1 A

Step navigation between screens

JavaScript
function navigateToStep(stepId) {
  // 1. Update page title
  document.title = stepTitles[stepId] + ' — Social Housing Application';

  // 2. Show new screen content
  showScreen(stepId);

  // 3. Move focus to new h1
  const heading = document.querySelector('#' + stepId + ' h1');
  if (heading) {
    heading.setAttribute('tabindex', '-1');
    heading.focus();
  }
}

Error summary with focus management

JavaScript
function handleFormSubmit(event) {
  event.preventDefault();
  const errors = validateForm();

  if (errors.length) {
    // Build error summary
    showErrorSummary(errors);

    // Move focus to summary
    const summary = document.querySelector('.error-summary');
    summary.focus();
  } else {
    navigateToStep('review');
  }
}

function showErrorSummary(errors) {
  const summary = document.querySelector('.error-summary');
  const list = summary.querySelector('ul');
  list.innerHTML = errors.map(err =>
    `<li><a href="#${err.fieldId}">${err.message}</a></li>`
  ).join('');
  summary.hidden = false;
}

Conditional field reveal

JavaScript
const employmentRadios = document.querySelectorAll('[name="employment"]');
const employerGroup = document.getElementById('employer-group');

employmentRadios.forEach(radio => {
  radio.addEventListener('change', () => {
    if (radio.value === 'employed' && radio.checked) {
      // Reveal conditional field
      employerGroup.hidden = false;

      // Move focus to the new field
      const input = employerGroup.querySelector('input');
      if (input) input.focus();
    } else {
      employerGroup.hidden = true;
    }
  });
});
